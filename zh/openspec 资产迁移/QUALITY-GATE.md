# 统一就绪门禁质量检查

本文件用于 `propose` 流程的最后一环，确保规划文档不仅覆盖率达标，而且具备实施落地性。

在 change 进入 `apply` 或实施阶段前，必须通过以下所有检查。

## 1. 架构权威检查
- [ ] 方案是否和主 spec / 架构权威保持一致？
- [ ] 是否明确了前后端或上游/下游的权威边界？
- [ ] 是否没有引入未声明的权威来源？

## 2. 需求明确性检查
- [ ] 是否在提案中明确列出了修改范围 (Scope)？
- [ ] 是否在提案或设计中明确列出了非目标 (Non-Goals)？
- [ ] 是否分析了潜在风险 (Risks) 及缓解策略？
- [ ] 是否提供了明确且可验证的验收标准 (Acceptance Criteria)？
- [ ] 是否存在“表述完整但不能落地”的模糊项？

## 3. 架构与接口影响评估
- [ ] 是否在设计中列出了受影响对象（文件、模块、接口）？
- [ ] 是否在设计中说明了 API / Contract 的变化影响？
- [ ] 是否在设计中详细说明了状态迁移影响？
- [ ] 是否在设计中说明了持久化影响？

## 4. 实施落地性检查
- [ ] 任务中的粒度是否达到“实现者无需再猜”的清晰度？
- [ ] 任务是否明确了每个任务对应的文件范围？
- [ ] 任务是否为每一步提供了具体、可执行的验证命令？
- [ ] 任务是否明确了 MVP 建议范围？
- [ ] 任务是否单列了 Foundational / Blocking 任务？
- [ ] 任务是否为每个 Slice 提供了独立验收标准？
- [ ] 任务是否标出了并行机会？
- [ ] 任务是否保留 `1.1 / 1.2` 风格，并支持 `[P]` 与 `[Slice-X]` 标签？

## 5. 归档安全检查
- [ ] 方案是否能在 archive 后安全地合并为正式主 spec？
