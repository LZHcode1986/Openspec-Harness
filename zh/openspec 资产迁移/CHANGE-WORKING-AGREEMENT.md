# 主动变更驱动开发契约（Active Change）

本文件定义 OpenSpec 在项目中的工作方式。目标是把 OpenSpec 作为变更工作台，而不是事后归档仓库。

## 1. 新建与变更规则
- 先规划后实施。
- 同一意图、同一主路径的收敛修正，优先更新当前 change。
- 需求边界变化明显时，必须新建 change。

## 2. 认知回写机制
- 实施过程中如果发现理解有偏差，优先回写到当前 change 的 proposal / design / specs / tasks。
- 不要只改代码而不回写文档。

## 3. Archive 时机定义
- `archive` 只用于 change 收口完成并准备并入正式规范库。
- 只有当以下条件全部满足时，才允许执行 archive：
  1. 该 change 对应的实现范围已经完成。
  2. 代码与 proposal / design / specs / tasks 已重新对齐。
  3. 所有承诺的验证命令已执行并通过。
  4. 实现中新发现的重要认知已经回写到 change 文档。
  5. 归档后主 spec 不会退化。

## 4. 实施阶段定位
- OpenSpec 负责规划、约束、追踪与归档安全。
- 代码编写、测试执行、交付规范仍由项目自己的实施流程负责。
